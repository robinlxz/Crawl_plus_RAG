# P1: RAG Debug & Visibility 增强实施计划

根据您的严格要求，我们将实施以下增强，重点在于**真实数据透传**和**全链路可观测性**。

## 1. 核心变更：Generator 返回结构增强
**目标**: 确保 UI 展示的是**真实**发送给 LLM 的 Prompt，而不是 UI 自己拼凑的副本。

修改 `src/generator/generate.py`:
*   **方法**: `RAGGenerator.answer(question, retrieved_chunks)`
*   **变更**: 返回值从单纯的 `str` (answer) 变更为 `Dict`:
    ```python
    {
        "answer": "LLM生成的回答...",
        "debug": {
            "final_messages": [...] # 包含 System, User (Context+Query) 的完整列表
        }
    }
    ```
*   **兼容性**: 这是一个 Breaking Change，需要同步修改 `web_ui.py` 的调用处。

## 2. 前端变更：Debug Console 集成
修改 `src/web_ui.py`，在助手回答下方增加折叠面板。

### Tab 1: 🔍 Retrieval (检索透视)
直接使用 `searcher.search()` 的原始返回结果，展示：
*   **Rank & Score**: 明确显示相似度分数。
*   **Full Content**: 使用 `st.text_area` 或 `st.code` 展示完整文本块，不截断，以便检查切分边界。
*   **Metadata**: 展示 `Block ID`, `URL`, `Title`。

### Tab 2: 🧠 Prompt (真实提示词)
从 `generator.answer()` 的返回结果中提取 `debug.final_messages` 并展示。
*   使用 `st.json` 展示结构化消息，清晰区分 System Prompt 和 User Context。

### Tab 3: ⏱️ Stats (性能瀑布流)
在 UI 层进行精确打点计时：
*   `Retrieval Latency`: 搜索耗时。
*   `Generation Latency`: LLM 调用耗时。
*   `Total Latency`: 端到端耗时。

## 3. 执行步骤
1.  **后端改造**: 修改 `src/generator/generate.py`，封装返回结果。
2.  **前端适配**: 修改 `src/web_ui.py`，适配新的返回结构，并增加计时逻辑和 Debug 组件渲染代码。
3.  **验证**: 启动服务，提问一个问题，检查 Debug Console 是否如预期显示。

此计划严格遵循“不改变检索/生成核心逻辑”的原则，仅做数据透传和展示。
